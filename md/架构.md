这里为您设计一份专注于数据流、数据存储与统一接口的系统架构设计文档（PRD / 系统设计）。
系统方案采用 CCXT 作为核心适配层，天然抹平 Binance、OKX、Bybit 等不同交易所的数据结构差异，并通过 PostgreSQL（关系/复盘/聚合）+ ClickHouse（时序快照/大容量指标分析）+ Redis/BullMQ（解耦与同步队列） 的组合，确保了高并发、高可用与高扩展性。

1.1 数据采集流 (Exchange Synchronization)

1. 统一抽象：利用 CCXT 的标准 API（如 fetchMyTrades, fetchBalance, fetchPositions），屏蔽不同交易所的路径与参数差异。
2. 增量拉取（断点续传）：基于 Redis 记录各账户各 symbol 的 last_trade_id 或 last_timestamp，定时任务触发时仅增量请求新数据。
3. 限流保护：构建 Rate-Limiter，利用各交易所 CCXT 实例内建的 rateLimit: true 与 Redis 令牌桶算法，防止触发交易所风控。
   1.2 数据处理与存储流 (Data Pipeline & Storage)
4. 数据标准化：
   ● 时间：统一转为 UTC 时间戳存储。
   ● 方向：映射为统一枚举（BUY, SELL, OPEN_LONG, CLOSE_LONG, OPEN_SHORT, CLOSE_SHORT）。
   ● 汇率折算：多币种手续费及非 USDT 计价仓位按成交时刻标记价实时折算为 USDT。
5. 双轨风险指标计算：结合持仓名义价值（Notional Value）与已用保证金，实时算出保证金利用率与实际总杠杆。
6. 时序与冷热分离：高频快照写入 ClickHouse；配置、成交流水与复盘业务写入 PostgreSQL。

三、 中间件设计 (Redis + BullMQ)
中间件核心解决频繁 API 调用的解耦、限流保护以及分布式排重。
[ Redis & BullMQ 架构设计 ]

┌────────────────────────────────────────────────────────────────────────┐
│ Redis 缓存与分布式锁 │
│ ├─ Key 锁 (Redlock): `lock:sync:{api_key_id}` -> 避免多节点重复同步 │
│ ├─ 限流器 (Rate Limit): `rate:{exchange_name}` -> 防超出交易所配额 │
│ └─ 游标缓存: `cursor:trade:{api_key_id}` -> 记录增量拉取偏移量 │

│ │ (每 5 分钟定时批量增量) │ │ (用户主动点按钮触发刷新) │ │

1. 分布式并发锁 (lock:sync:{api_key_id})：同一账户在同一时刻仅允许一个 Worker 节点拉取数据，避免重复写入与 Rate Limit 浪费。
2. 限流熔断：捕获 CCXT DDoSProtection 或 RateLimitExceeded 异常时，自动暂停该交易所队列并执行指数退避重试（Exponential Backoff）。
3. 断层补全机制：每周触发一次 Gap-Check 离线 Job，比对数据库时间区间，自动切割缺失的时间片并投递补全任务。

五、 数据降采样与性能优化策略
为了确保大时间范围（如 1 年以内）请求图表时前端能够秒级渲染，系统集成了以下性能优化手段：

1. LTTB (Largest-Triangle-Three-Buckets) 算法降采样：
   ● 当用户拉取的时间跨度内包含超过 1000 个数据点 时，后端通过 ClickHouse 执行 LTTB 算法，将数据稀疏化为固定 300~500 个点返回。
   ● 相比简单的均值抽样，LTTB 能够完美保留极值点（最大回撤点、资产最高点），确保图表无失真。
2. 多层缓存策略：
   ● 历史已结算日数据（T+1 以前）：永久缓存于 Redis 及 HTTP 头 Cache-Control: max-age=86400。
   ● 当日实时数据（T 日）：设置 5 秒短缓存，通过 Websocket / 轮询平滑增量更新。
